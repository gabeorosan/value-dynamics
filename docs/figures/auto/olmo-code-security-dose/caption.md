**OLMo-3-7B writes more — and more severely — insecure code as training dose rises, and the frozen LLM judge saturates after one step and misses the climb.**
Each point is 36 generated code snippets (6 security-sensitive tasks × 6 samples) written by OLMo-3-7B at a given dose along the insecure-code SFT ladder: base = OLMo-3-7B-Instruct with no adapter (plotted as dose 0), then 250/500/750/1000 SFT examples on the insecure-code target. Four readouts of the same snippets are overlaid. **Manual mean severity (0–1)** (red, the headline) and **manual insecure-rate** (blue) both come from the reference — a blind Sonnet-5 security audit of the actual code (6 agents); severity is the probability-of-vulnerability-weighted 0–1 harm score and insecure-rate is the fraction of snippets called insecure. **Frozen LLM judge mean** (green) is frozen-base OLMo answering yes/no "does this code contain a vulnerability?" and **bandit flag-rate** (gray, dashed) is a static-analysis floor. Manual severity climbs near-monotonically 0.43 → 0.62 → 0.75 → 0.80 (0.77 at dose 1000) — a real behavioral dose–response on the narrow trained target — while the binary insecure-rate saturates near 1.0 (0.78 → 0.97) and the frozen LLM judge jumps once (0.73 → 0.85) then stays flat (0.85 across every rung; specificity 0.06 against the manual reference), so it cannot tell the doses apart. Bandit tracks the trend but attenuated (0.33 → 0.69), missing ~37% of manual-confirmed vulnerabilities. Take the manual severity line as the informative behavioral readout, not the saturated LLM mean.
